# SWML Orchestrator Architecture (Claude 4.6 Edition)

## 1. 哲学: "Differentiable Context over Differentiable Weights"
（重みの更新より、文脈の更新を）

強化学習（RL）を用いてLLMのパラメータ自体を更新するアプローチは、膨大な計算資源を要求します。
本アーキテクチャでは、**「Claude 4.6の圧倒的なゼロショット推論力」**と**「SWML理論（情報重力）」**を組み合わせることで、**パラメータを固定したまま、コンテキスト（システムプロンプト）を自動進化させる**ことでエージェントを成長させます。

林さんの「言葉の力学」に基づき、環境からの摩擦（エラー、指摘）を「新たなルール」としてナレッジベースに物理的に蓄積し、次回の推論時に強制注入します。

---

## 2. コア・コンポーネント

### A. Task Algebra Engine (タスク分解・ルーター)
- **役割:** ユーザーからの曖昧な指示（Intent）を受け取り、SWMLのタスク代数（§7）に基づき、実行可能な最小単位のサブタスクに分解する。
- **実装:** ユーザーの入力 → Claude 4.6によるタスクツリー（JSON）の生成。

### B. Context Assembler (情報重力の制御器)
- **役割:** エージェントが行動（Interaction）を起こす直前に、**「そのタスクに必要な過去の失敗録、SOP、林さんのコーディング規約」**をナレッジベースから抽出し、システムプロンプトに動的に合体させる。
- **実装:** Markdownファイルのタグ検索、または LanceDB（ベクトル検索）による関連コンテキストのRAG。

### C. $\Omega$ Feedback Loop (自動学習・反省ループ)
- **役割:** 世界（World）に干渉した結果生じた「摩擦（CIエラー、PRレビューでのダメ出し）」を検知し、恒久的なルールとして昇華させる。
- **実装:** 
  1. GitHub WebhookでエラーログやPRコメントを受信。
  2. Claude 4.6が「なぜ失敗したのか」「次はどうすべきか」を抽出（Reflection）。
  3. `KNOWLEDGE.md` または特定のナレッジディレクトリに「新しい教訓（Lesson Learned）」として自動追記（ファイル書き込み）。

---

## 3. データフロー（The SWML Loop）

```text
[1. Intent (GitHub Issue等)]
       ↓
[2. Context Assembly] ← (ナレッジベースから過去の教訓をPull)
       ↓
[3. Inference (Claude 4.6 Opus/Sonnet)]
       ↓
[4. World Execution] (コード実行、PR作成)
       ↓
[5. Result Evaluation] (CIテスト、人間のレビュー)
       ├─ [成功] → タスク完了
       └─ [失敗/指摘] → [6. Knowledge Extraction]
                             ↓
                      (ナレッジベースへPush/自動追記)
```

## 4. この設計の優位性
1. **圧倒的な低コスト・即時性:** GPUクラスタが不要。APIを叩くだけで「秒」で賢くなる。
2. **忘却の防止:** LLMのコンテキストウィンドウがリセットされても、ファイル（物理）として教訓が残るため、絶対に同じミスを繰り返さない。
3. **人間による可読性:** エージェントが何を学んだかが「Markdownのテキスト」として残るため、林さんが直接編集して軌道修正することが容易（ブラックボックス化しない）。
